## Medium Priority
4. **Wash sale / tax columns in Holdings** — Add toggleable tax columns: adjusted cost basis (including disallowed wash sale loss carried over) and a "Carried Disallowed Loss" column. Open P&L remains based on nominal cost basis and market value — tax columns are informational overlay only.
5. **Transaction timestamps with timezone** — Show full time (not just date) for each transaction, displayed in the original timezone of the transaction (EST/EDT as provided in the CSV).
6. **Beta expectation deviation diagnostic** — Add a Risk/Attribution-style view that answers: "Why did my portfolio outperform or underperform what its beta implied today?" The current Attribution card explains each ticker's realized contribution to portfolio return, but it does not separate expected beta-driven movement from idiosyncratic movement. For each holding, compute expected return as `holding_beta_vs_benchmark * benchmark_return` over the selected window/day, compare it to the holding's actual return, and show the residual/deviation. Weight that residual by portfolio weight to estimate each ticker's contribution to portfolio-level beta surprise. UI should highlight tickers that materially overperformed or underperformed their beta expectation, show whether the portfolio's total return was explained by market beta vs stock-specific residuals, and include clear labels such as "Expected from beta", "Actual return", "Residual", and "Portfolio impact". Start with daily mode using the selected benchmark (SPY/QQQ/IWM/DIA), then optionally support date ranges. Backend likely belongs near `risk_engine.py` or `attribution.py`, exposed as a typed endpoint consumed through `frontend/src/api/client.ts`. Important edge cases: insufficient history for a ticker beta, new positions with short windows, zero/near-zero benchmark variance, missing prices from yfinance, and cash weight contributing zero beta/zero return. Acceptance criteria: a user can look at a bad or unusually good portfolio day and identify which holdings merely behaved as beta predicted versus which holdings materially deviated above or below expectation.

## Future Phases
9. **Stock split handling** — Webull CSVs do not retroactively adjust pre-split transactions. The engine's `walk_transactions_avg_cost` sees original pre-split quantities/prices while yfinance returns split-adjusted prices, causing incorrect avg cost and unrealized P&L for any position bought before a split. Fix: detect splits from yfinance (already fetched in `portfolio_engine.build()`) and apply the ratio to pre-split `avg_cost` and `shares_held` entries after the accounting walk, before they're used in `build_summary` and `build_holdings`.


7. **Options trades support** — Partially Done. Options CSVs share the same structure. Parse and display options transactions, P&L, and exposure. Separate phase.
8. **Holdings time travel** — Date picker on the Holdings page lets the user rewind to any past date (clamped between first transaction date and today). Backend replays all transactions up to that date, fetches prices as of market close, and returns holdings/PnL/weights as they stood then. Frontend shows a banner when in time-travel mode; resets to current on tab switch or page reload (ephemeral UI state only, no persistence). Backend approach: add `?as_of=YYYY-MM-DD` param to `GET /portfolio/holdings`.
